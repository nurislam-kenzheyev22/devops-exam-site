Q12: Why was Gradle chosen instead of Maven?

Answer (English):

Gradle was chosen for its flexibility, faster builds through incremental compilation, Groovy/Kotlin DSL for readable build scripts, better dependency management, and parallel task execution. It's more modern and suitable for complex projects compared to Maven's XML-based configuration.

Q13: What is the Gradle Wrapper and why is it important?

Answer (English):

The Gradle Wrapper (`gradlew`) is a script that ensures everyone uses the same Gradle version. It automatically downloads the correct Gradle version if missing, eliminating "works on my machine" problems. The wrapper files (`gradlew`, `gradlew.bat`, `gradle/wrapper/`) are committed to Git.

Q14: Where does Gradle store build artifacts?

Answer (English):

Gradle stores build artifacts in the `build/libs/` directory. The JAR file is typically named `{project-name}-{version}.jar` (e.g., `todo-0.0.1-SNAPSHOT.jar`). This location is standardized and used by Docker and deployment tools.

Q15: Why is reproducible build important in DevOps?

Answer (English):

Reproducible builds ensure that the same source code always produces the same binary output, regardless of when, where, or by whom it's built. This enables reliable deployments, debugging, security auditing, and ensures that what works in development also works in production.

Q16: What is the difference between `gradle build` and `./gradlew build`?

Answer (English):

`gradle build` uses the Gradle installation on your system (may vary between machines). `./gradlew build` uses the Gradle Wrapper, ensuring everyone uses the exact same Gradle version specified in the project. The wrapper is preferred for consistency and reproducibility.
