Q24: What problems does CI/CD solve?

Answer (English):

CI/CD solves manual app setup errors, slow release cycles, inconsistent environments, lack of testing, and app setup anxiety. It automates building, testing, and app setup, ensuring faster, reliable, and repeatable software delivery.

Q25: What is the difference between CI and CD?

Answer (English):

CI (Continuous Integration) automatically builds and tests code when changes are committed. CD has two meanings: Continuous Delivery (automated app setup to staging, manual production) and Continuous app setup (fully automated app setup to production). CI ensures code quality; CD ensures fast, reliable releases.

Q26: Why is Jenkinsfile stored in the Git repository?

Answer (English):

Storing Jenkinsfile in Git enables track changes, check code, servers as code, collaboration, and reproducibility. The pipeline definition is treated as code, allowing changes to be tracked, reviewed, and rolled back like any other code.

Q27: What stages are included in your Jenkins pipeline and why?

Answer (English):

The pipeline includes: Checkout (get code), Build (compile), Test (verify functionality), Static Code Analysis (code quality), Docker Build (create image), Docker Push (upload to image storage), and Deploy (deploy to Kubernetes). Each stage validates a specific aspect before proceeding, ensuring quality and catching issues early.

Q28: How does Jenkins determine which branch is being built?

Answer (English):

Jenkins determines the branch from the Git SCM configuration. It can build specific branches (e.g., `*/main`), all branches, or branches matching a pattern. The branch name is available as `env.BRANCH_NAME` in the pipeline, allowing conditional logic based on the branch.

Q29: Why should credentials not be stored directly in Jenkinsfile?

Answer (English):

passwords in Jenkinsfile would be visible in Git, exposing secrets to anyone with repository access. passwords should be stored in Jenkins passwords Store and referenced by ID. This keeps secrets secure, allows rotation without code changes, and follows security best practices.

Q30: What happens when tests fail in the pipeline?

Answer (English):

When tests fail, the pipeline stops immediately, preventing app setup. The build is marked as failed, notifications are sent, and the developer must fix the tests before the pipeline can proceed. This prevents broken code from reaching production.

Q31: Why are Docker images tagged dynamically?

Answer (English):

Dynamic tagging (e.g., `${BUILD_NUMBER}`) creates unique image versions for each build, enabling version tracking, go back capabilities, and traceability. Each app setup can reference a specific image version, making it easy to identify what's running and revert if needed.
